## 1. 프로젝트 개요

### 1.1 프로젝트명

**MathLens AI** — 손글씨 수학 풀이 인식 기반 개인 맞춤형 AI 수학 튜터 모바일 애플리케이션

### 1.2 플랫폼

- iOS (iPhone, iPad) — iOS 16.0 이상
- Android — Android 10 (API Level 29) 이상

### 1.3 핵심 가치 제안

| 구분 | 설명 |
|------|------|
| 풀이 인식 | 사용자가 펜 또는 손가락으로 입력한 수학 풀이 과정을 실시간 인식 |
| 개념 이해도 추론 | 인식된 풀이에서 학습자가 올바른 개념을 이해하고 있는지 AI가 판단 |
| 오답 원인 분석 | 단순 정오 여부가 아닌 오답의 근본 원인(개념 오류, 계산 실수 등)을 추론 |
| 맞춤 설명 | 오답 유형에 따른 개인화된 개념 재설명 제공 |
| 반복 학습 | 동일 유형의 문제를 재출제하여 완전 학습(Mastery Learning) 유도 |

---

## 2. 배경 및 목적

### 2.1 시장 배경

- 국내 에듀테크 시장은 2024년 약 7조 원 규모로 매년 15% 이상 성장 중
- 수학은 학습 격차가 가장 크게 발생하는 과목으로, AI 기반 개인 맞춤 학습 수요 급증
- 기존 수학 앱은 정답 여부만 판별하며, **풀이 과정 분석 기반 피드백**은 시장 내 미충족 영역
- 손글씨 인식 기술(HTR, Handwritten Text Recognition)과 멀티모달 LLM의 발전으로 구현 가능성 확보

### 2.2 프로젝트 목적

1. 학습자가 종이에 쓰듯 자연스럽게 스마트 기기 위에서 수학 풀이를 작성
2. AI가 풀이 과정을 단계별로 분석하여 **어느 단계에서 어떤 개념이 틀렸는지** 진단
3. 학습자의 오개념을 정확히 짚어주는 **맞춤형 개념 설명** 제공
4. 같은 유형의 문제를 반복 출제하여 **완전 학습** 달성

### 2.3 타깃 사용자

| 타깃 | 세부 내용 |
|------|-----------|
| 주요 타깃 | 중학교 1학년 ~ 고등학교 3학년 (13~19세) |
| 보조 타깃 | 수능 재수생, 대학 기초 수학 수강생 |
| 보호자 | 학습 현황 모니터링을 원하는 학부모 |

## 3. 핵심 기능 요구사항

### 3.1 펜 입력 및 손글씨 인식 (Handwriting Recognition)

#### 3.1.1 입력 방식
- 스타일러스 펜 (Apple Pencil, S-Pen 등) 및 손가락 터치 모두 지원
- 실시간 잉크 렌더링: 필압 및 속도에 반응하는 자연스러운 필기감
- 수식 전용 인식 엔진 연동 (수학 기호, 분수, 제곱근, 적분 기호 등 포함)

#### 3.1.2 인식 범위
- 기본 사칙연산 및 수식 표현
- 방정식, 부등식, 함수식
- 기하학적 도형 및 보조선 스케치
- 행렬, 집합, 수열, 극한, 미적분 기호
- 한글/영문 풀이 설명 텍스트 병행 인식

#### 3.1.3 인식 정확도 요구
- 수식 인식 정확도: **95% 이상** (표준 필기체 기준)
- 인식 응답 시간: 풀이 완료 후 **3초 이내** 분석 시작

---

### 3.2 AI 풀이 분석 엔진

#### 3.2.1 풀이 단계 분해
- 인식된 풀이를 논리적 단계로 자동 분리
- 각 단계의 수학적 유효성 검증
- 단계 간 논리적 연결성 분석

#### 3.2.2 개념 이해도 추론
AI는 다음 항목을 기준으로 학습자의 개념 이해 수준을 판별한다:

| 판별 항목 | 설명 |
|-----------|------|
| 공식 적용 | 해당 문제 유형에 맞는 공식을 사용했는가 |
| 풀이 방향 | 접근 전략이 수학적으로 올바른가 |
| 중간 계산 | 각 단계의 계산 과정이 정확한가 |
| 논리 흐름 | 단계 간 논리적 비약 또는 오류가 없는가 |
| 최종 검증 | 답을 검증하는 과정을 시도했는가 |

#### 3.2.3 오답 원인 분류 체계

```
오답 유형
├── 개념 오류 (Conceptual Error)
│   ├── 공식 혼동 (예: 인수분해 공식 오적용)
│   ├── 개념 미형성 (예: 함수의 정의 미이해)
│   └── 유사 개념 혼동 (예: 순열/조합 혼동)
├── 절차 오류 (Procedural Error)
│   ├── 풀이 순서 오류
│   ├── 이항 부호 실수
│   └── 단계 누락
├── 계산 실수 (Computational Error)
│   ├── 산술 계산 오류
│   └── 분수/소수 처리 실수
└── 문제 이해 오류 (Comprehension Error)
    ├── 조건 미반영
    └── 구하는 것 오해
```

---

### 3.3 맞춤형 피드백 및 개념 설명

#### 3.3.1 피드백 구조
1. **오류 위치 하이라이팅**: 틀린 단계를 시각적으로 표시
2. **오류 유형 안내**: 어떤 종류의 오류인지 친절하게 설명
3. **개념 재설명**: 관련 개념을 학년 수준에 맞춰 재설명
4. **올바른 풀이 예시**: 정석 풀이 단계별 제시 (애니메이션 옵션)
5. **핵심 공식 카드**: 관련 공식과 예제를 카드 형태로 제공

#### 3.3.2 설명 수준 조절
- 학습자의 학년, 이전 학습 이력, 오답 패턴을 반영하여 설명 깊이 자동 조절
- 쉬운 말로 설명 / 수학적 표현으로 설명 선택 옵션 제공

#### 3.3.3 멀티미디어 지원
- 개념 설명 텍스트 + 수식 + 그래프/도형 시각화
- 개념 설명 TTS(Text-to-Speech) 지원 (한국어)

---

### 3.4 반복 학습 시스템

#### 3.4.1 유사 문제 생성
- 오답 문제와 동일한 개념·유형이지만 숫자·조건이 다른 새 문제 자동 생성
- 난이도를 점진적으로 상승시키는 스캐폴딩(Scaffolding) 방식 적용
- 문제 생성 시 교육과정(국내 2015/2022 개정 교육과정) 범위 준수

#### 3.4.2 완전 학습 판별
- 동일 유형 연속 3회 정답 → 해당 개념 "습득 완료" 처리
- 습득 완료 개념은 일정 기간 후 복습 문제로 재등장 (망각 곡선 기반 간격 반복)

#### 3.4.3 학습 진행 추적
- 개념 단위 습득 현황 시각화 (개념 맵)
- 오답 노트 자동 생성 및 관리
- 주간/월간 학습 리포트

---

### 3.5 문제 입력 방식

| 입력 방식 | 설명 |
|-----------|------|
| 카메라 촬영 | 교재, 시험지 문제를 카메라로 촬영하여 업로드 |
| 갤러리 이미지 | 저장된 문제 이미지 업로드 |
| 앱 내 문제은행 | 과목/단원/난이도별 문제 선택 |
| 직접 입력 | 앱 내 수식 에디터로 문제 직접 작성 |

- 취약 개념 리스트

## 6. 손글씨 인식 기술 상세 요구사항

*(v1.1에서 추가된 섹션 — 내용 동일, 변경 없음)*

본 챕터는 MathLens AI의 핵심 기술인 손글씨 수식 인식(HTR) 구현에 필요한 기술적 방법론, 도구, 구현 전략을 상세히 기술한다. 제안사는 아래 내용을 기반으로 자사의 구체적인 구현 방안을 제안서에 포함해야 한다.

### 6.1 손글씨 인식 기술 개요 및 접근 방법

| 접근 방법 | 설명 | 장점 | 단점 |
|-----------|------|------|------|
| 온라인 인식 (Online HTR) | 펜의 이동 경로(스트로크) 데이터를 실시간으로 분석 | 빠른 응답, 필순 정보 활용 가능 | 스타일러스 펜 필수 |
| 오프라인 인식 (Offline HTR) | 완성된 이미지(사진/스캔)를 분석 | 카메라 촬영 지원, 범용성 높음 | 이미지 품질에 민감 |
| 하이브리드 | 온라인 + 오프라인 병행 | 두 방식의 장점 결합 | 구현 복잡도 증가 |

**본 프로젝트의 기본 방향은 하이브리드 방식**으로, 스타일러스 펜 입력 시 온라인 인식, 카메라 촬영 시 오프라인 인식을 적용한다.

> *(이하 섹션 6.2 ~ 6.8은 RFP v1.1 문서를 참조)*

## 7. 서버-모바일 통신 및 AI 추론 설계 요구사항

> **v1.2 신규 챕터**  
> 본 챕터는 백엔드 서버와 모바일 앱 간 문제 전달 방식, 펜 입력 AI 인가 후 서버 전송 방식, 풀이 과정 점검 파이프라인, 개념 부족 추론 및 맞춤 질문·설명 설계에 대한 상세 기술 요구사항을 정의한다. 제안사는 이 챕터의 각 항목에 대한 구체적인 구현 방안을 제안서에 필수로 포함해야 한다.

### 7.1 백엔드–모바일 문제 전달 요구사항

#### 7.1.1 문제 전달 프로토콜

제안사는 다음 두 가지 통신 방식을 모두 지원하는 설계를 제시해야 한다:

**① REST API (기본 문제 로딩)**

| 항목 | 요구사항 |
|------|---------|
| 프로토콜 | HTTPS (TLS 1.3) REST |
| 인증 | JWT Bearer Token (RS256 알고리즘, 만료 1시간) |
| 응답 형식 | JSON (`application/json`) |
| 캐시 정책 | `Cache-Control: max-age=3600` |
| 응답 시간 | p95 기준 300ms 이내 |
| 문제 스키마 | 문제 ID, 수식(LaTeX), 메타데이터(학년·단원·난이도·필요개념), 힌트, 정답 해설 포함 |

**② WebSocket (실시간 적응형 문제 추천 — Phase 2)**

| 항목 | 요구사항 |
|------|---------|
| 프로토콜 | WSS (WebSocket Secure) |
| 연결 유지 | Heartbeat 30초 주기 |
| 이벤트 유형 | `session_init`, `problem_push`, `heartbeat`, `session_end` |
| 용도 | 세션 중 풀이 완료 즉시 다음 문제 적응형 추천 |

#### 7.1.2 오프라인 지원 요구사항

- 앱 실행 시 최소 **10개 문제**를 기기 로컬 저장소에 선캐시
- 네트워크 미연결 상태에서도 캐시된 문제로 학습 세션 진행 가능
- 오프라인 중 제출된 풀이는 큐에 보관 후 재연결 시 자동 서버 전송
- 로컬 캐시 암호화 필수 (AES-256)

#### 7.1.3 문제 추천 알고리즘 요구사항

서버는 다음 요소를 복합 반영한 문제 추천 로직을 구현해야 한다:

```
추천 점수 = (개념_부족도 × 0.4)
           + (미학습_문제_우선도 × 0.3)
           + (간격_반복_만료도 × 0.2)
           + (난이도_적정도 × 0.1)
```

- **개념 부족도**: 최근 오답에서 추론된 취약 개념과 연관된 문제 우선 추천
- **간격 반복**: Ebbinghaus 망각 곡선 기반 복습 시점(3일→7일→14일→30일) 자동 계산
- **난이도 조절**: 최근 3회 연속 정답 시 다음 단계 난이도 문제 제시

### 7.2 펜 입력 정답 전송 설계 요구사항

#### 7.2.1 온디바이스 AI 인가 (1차 처리)

모바일 기기 내에서 실행되는 경량 HTR 모델이 펜 입력을 1차 처리한다:

| 항목 | 요구사항 |
|------|---------|
| 추론 시간 | < 200ms (최신 플래그십 기기 기준) |
| 모델 크기 | iOS Core ML: 15MB 이하 / Android TFLite INT8: 20MB 이하 |
| 출력 | LaTeX 변환 결과 + 토큰별 신뢰도 점수 (0.0 ~ 1.0) |
| 신뢰도 임계값 | 0.85 이상: 자동 인가 / 0.60~0.85: 사용자 확인 / 0.60 미만: 재입력 요청 |

**인가 처리 흐름:**

```
[펜 스트로크 수집 완료]
        │
        ▼
[온디바이스 HTR 모델 추론]
        │
        ├─ confidence ≥ 0.85 → 인가 완료, 제출 버튼 활성화
        ├─ 0.60 ≤ confidence < 0.85 → "인식된 수식이 맞나요?" 확인 UI 표시
        └─ confidence < 0.60 → "다시 써주세요" 안내 + 캔버스 초기화 제안
```

#### 7.2.2 서버 전송 페이로드 구조

클라이언트는 AI 인가 완료 후 다음 구조의 데이터를 서버에 전송해야 한다:

```
제출 페이로드 구성요소:
1. submission_id      : 클라이언트 생성 UUID (중복 제출 방지)
2. problem_id         : 해당 문제 ID
3. user_id / session_id
4. device_info        : 플랫폼, OS 버전, 기기 모델, 입력 장치 종류
5. handwriting_data   : 캔버스 이미지 URL + 스트로크 InkML URL (S3 선업로드)
6. client_recognition : 온디바이스 변환 LaTeX + 단계별 신뢰도 + 인가 통과 여부
7. time_spent_sec     : 풀이 소요 시간
8. hints_used         : 사용한 힌트 횟수
```

#### 7.2.3 이미지/잉크 파일 전송 방식

네트워크 효율을 위해 이미지 및 잉크 파일은 S3 Pre-signed URL을 통해 클라이언트가 직접 업로드한다:

```
Step 1: POST /api/v1/submissions/upload-url
        → 서버가 S3 Pre-signed URL 반환 (유효시간 5분)

Step 2: PUT <presigned_url>
        → 클라이언트가 캔버스 이미지 및 InkML 파일 직접 S3 업로드

Step 3: POST /api/v1/submissions
        → 페이로드 전송 (이미지 URL 참조 포함)
```

#### 7.2.4 서버 수신 후 HTR 재검증

서버는 클라이언트의 LaTeX 인식 결과를 재검증해야 한다:

- MathPix API 또는 자체 호스팅 Pix2tex로 캔버스 이미지 독립 처리
- 클라이언트 결과와 서버 결과를 **신뢰도 가중 앙상블**로 통합
- 불일치율 > 20% 시 서버 결과를 우선 채택
- 양측 결과 모두 confidence < 0.70 시 사용자에게 재제출 요청

### 7.3 문제 풀이 과정 점검 파이프라인 요구사항

제안사는 다음 5단계 파이프라인을 모두 구현해야 한다:

#### 7.3.1 파이프라인 단계 정의

```
Step 1: HTR 재검증 및 LaTeX 정규화
        ─────────────────────────────
        • 서버측 독립 HTR 처리
        • 앙상블 통합 → 최종 LaTeX 확정
        • SymPy 파싱 가능 여부 검증
        목표 처리 시간: < 1s

Step 2: 풀이 단계 파서
        ─────────────────────────────
        • 개행/등호/∴ 기준 단계 분리
        • 단계 유형 분류: equation / expression / conclusion / text / figure
        • 각 단계 SymPy 수식 파싱
        목표 처리 시간: < 500ms

Step 3: 수학적 타당성 검증
        ─────────────────────────────
        • 인접 단계 간 등가 변환 검증 (수치 대입 랜덤 검사 5회)
        • 부호 오류, 이항 오류, 계수 오류 자동 탐지
        • 최종 답을 원식에 대입하여 검산
        목표 처리 시간: < 1s

Step 4: LLM 심층 분석
        ─────────────────────────────
        • 입력: 문제 + 정규화 풀이 + 검증 결과 + 학습 이력
        • 모델: GPT-4o 또는 Claude Sonnet (Fallback 설정 필수)
        • 출력: 구조화 JSON (오류 목록, 개념 태그, 취약 개념, 튜터링 계획)
        목표 처리 시간: < 5s

Step 5: 결과 저장 및 응답
        ─────────────────────────────
        • PostgreSQL 분석 결과 저장
        • 개념 마스터리 상태 업데이트
        • WebSocket 또는 REST 응답으로 클라이언트 전달
```

#### 7.3.2 실시간 진행 상황 스트리밍

분석 처리 중 클라이언트에 단계별 진행 상황을 WebSocket으로 전달해야 한다:

```json
{ "event": "htr_complete",     "progress": 20, "message": "수식 인식 완료" }
{ "event": "parsing_complete", "progress": 40, "message": "풀이 단계 분석 완료" }
{ "event": "verify_complete",  "progress": 60, "message": "수학적 검증 완료" }
{ "event": "llm_complete",     "progress": 85, "message": "AI 분석 완료" }
{ "event": "analysis_done",    "progress": 100,"result_url": "/api/v1/submissions/{id}/result" }
```

### 7.4 개념 부족 추론 및 맞춤 질문·설명 엔진 요구사항

#### 7.4.1 수학 개념 지식 그래프 구축

제안사는 다음 요건을 충족하는 개념 지식 그래프를 구축해야 한다:

| 항목 | 요구사항 |
|------|---------|
| 범위 | 2022 개정 교육과정 수학 전 범위 (중1 ~ 고3) |
| 노드 수 | 최소 300개 개념 노드 |
| 엣지 유형 | `prerequisite` (선수 학습), `related` (연관 개념) |
| 저장소 | Neo4j 또는 PostgreSQL JSONB + 재귀 CTE |
| 각 개념 노드 포함 정보 | 개념명, 교육과정 코드, 학년, 선수개념 ID 목록, 마스터리 임계값, 관련 오류 패턴 |

**개념 지식 그래프 예시 (이차함수 영역):**

```
[수의 계산] → [다항식 전개] → [인수분해]
                                    │
                          ┌─────────┴─────────┐
                          ▼                   ▼
                    [완전제곱식]        [이차방정식]
                          │                   │
                          └─────────┬─────────┘
                                    ▼
                          [이차함수의 그래프]
                                    │
                                    ▼
                          [이차함수의 꼭짓점]
                                    │
                                    ▼
                          [최댓값·최솟값 문제]
```

#### 7.4.2 개념 부족 추론 알고리즘

서버는 다음 5단계 추론 과정을 구현해야 한다:

```
1. 오류 발생 단계의 수학적 조작 식별
   └─ "완전제곱식 변환 단계에서 상수항 처리 누락"

2. 오류 → 개념 태그 매핑 (LLM 출력 + 규칙 기반)
   └─ "완전제곱식" → concept_id: C040

3. 선수 개념 역추적 (개념 그래프 BFS)
   └─ C040의 모든 선수 개념 트리 탐색

4. 학습 이력 교차 검증
   └─ 선수 개념들의 마스터리 상태 조회 → 미습득 개념 목록 생성

5. 우선순위 산출
   우선순위 점수 = (최근 오답 빈도 × 0.4)
                + (그래프 깊이 근접도 × 0.3)
                + (미학습 기간 × 0.2)
                + (관련 문제 빈출도 × 0.1)
```

#### 7.4.3 맞춤 질문·설명 생성 요구사항

**소크라테스 모드 (기본값):**
- 학생에게 정답을 직접 제공하지 않고, 유도 질문으로 스스로 발견하게 함
- 최대 3회 힌트 제공 후 3회 모두 실패 시 직접 설명 모드 자동 전환
- 각 질문은 단일 개념에 집중, 복수 개념 동시 질문 금지

**직접 설명 모드 (3회 실패 시 또는 학생 선택 시):**
- 개념 정의 → 핵심 공식 → 예제 2개 → 오류 수정 설명 순서 고정
- 설명 수준: `basic` / `medium` / `advanced` / `exam` 4단계 자동 선택 (학년 기반)

**대화 유지 요구사항:**
- 튜터링 대화는 세션 단위로 저장 (session_id 연계)
- 학생 응답 후 1.5초 이내 다음 질문 또는 피드백 생성
- 동일 개념에 대한 반복 오류 시 다른 접근법(예시, 시각화, 역산)으로 전환

#### 7.4.4 마스터리 상태 관리

| 이벤트 | 처리 |
|--------|------|
| 정답 (오류 없음) | 해당 개념 정답 스트릭 +1 |
| 오답 또는 오류 | 해당 개념 정답 스트릭 리셋 |
| 연속 3회 정답 | 마스터리 달성, `mastered=true`, 간격 반복 스케줄 생성 |
| 복습 주기 도래 | 해당 개념 복습 문제 오늘의 추천에 자동 추가 |
| 복습 실패 | 마스터리 해제, 재학습 큐 투입 |

#### 7.4.5 LLM 프롬프트 설계 요구사항

제안사는 다음 두 가지 LLM 프롬프트 모듈을 각각 설계하고 제안서에 명시해야 한다:

**① 풀이 분석 프롬프트**
- 입력: 문제 + 학생 풀이 (단계별) + 검증 결과 + 학습 이력
- 출력 (JSON): 단계별 오류 여부, 오류 유형, 개념 태그, 취약 개념 우선순위, 튜터링 계획
- 출력 토큰: 최대 2,000 토큰 (비용 최적화)
- Fallback: 주 모델 실패 시 대체 모델 자동 전환

**② 튜터링 질문 생성 프롬프트**
- 입력: 취약 개념, 오류 설명, 학년/수준, 현재 대화 단계, 이전 학생 답변
- 출력 (JSON): `question` (소크라테스 질문), `hint_level` (0~2), `switch_to_explain` (boolean)
- 질문 생성 시간: < 1.5s (p95)
