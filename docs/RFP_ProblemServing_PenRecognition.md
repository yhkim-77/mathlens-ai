## 1. 제안 배경 및 목적

### 1.1 배경

MathLens AI는 학습자가 스마트 기기 위에 펜으로 수학 풀이를 작성하면 AI가 이를 인식·분석하여 오답 원인과 맞춤 피드백을 제공하는 모바일 애플리케이션입니다.

현재 시스템의 두 가지 핵심 과제는 다음과 같습니다.

1. **문제 출제 파이프라인**: 백엔드 서버가 학습자 수준, 취약 개념, 학습 이력을 기반으로 적절한 문제를 선정·생성·전달하는 체계
2. **펜 입력 AI 인식 파이프라인**: 사용자가 펜으로 작성한 답안 이미지를 수집하여 AI 엔진이 정확히 인식·해석하는 체계

두 파이프라인은 독립적으로 동작하되 유기적으로 연동되어야 하며, 본 RFP는 이 두 영역의 설계·구현·통합을 위한 기술 제안을 요청합니다.

### 1.2 목적

| 번호 | 목적 |
|------|------|
| 1 | 백엔드에서 학습자 맞춤형 문제를 자동 출제하고 클라이언트에 전달하는 서비스 구현 |
| 2 | 사용자의 펜 입력(스타일러스·터치)을 안정적으로 캡처하고 AI 엔진에 전달하는 클라이언트-서버 인터페이스 설계 |
| 3 | HTR(Handwritten Text Recognition) 및 멀티모달 LLM을 연동하여 손글씨 수학 풀이를 고정밀로 인식하는 AI 파이프라인 구축 |
| 4 | 전체 흐름의 응답 지연, 정확도, 확장성에 대한 SLA 충족 |

## 2. 범위 정의

### 2.1 본 RFP의 대상 범위

```
[백엔드 문제 출제 서비스]          [클라이언트 펜 입력]
        │                                 │
        ▼                                 ▼
  문제 선정·생성·전달             잉크 캡처·이미지 변환
        │                                 │
        └──────────────┬──────────────────┘
                       ▼
              [AI 인식 파이프라인]
         이미지 전처리 → HTR → LaTeX 변환
                       │
                       ▼
              [풀이 분석 및 피드백]
              (본 RFP 범위 외)
```

**포함 범위**
- 문제은행 DB 설계 및 문제 출제 API
- 학습자 맞춤 문제 추천 로직
- LLM 기반 유사 문제 동적 생성
- 모바일 클라이언트의 펜 입력 캔버스 구현 (iOS PencilKit / Android Ink API)
- 손글씨 이미지 캡처, 전처리, 서버 전송 모듈
- HTR 엔진 연동 및 LaTeX 변환 서비스
- 인식 결과 저장 및 분석 엔진 전달 인터페이스

**제외 범위**
- 오답 원인 분석 엔진 (별도 RFP)
- 맞춤형 피드백 생성 (별도 RFP)
- 학습 리포트 및 대시보드 (별도 RFP)

## 3. 백엔드 문제 출제 시스템 요구사항

### 3.1 문제은행 데이터베이스

#### 3.1.1 문제 메타데이터 스키마

각 문제는 다음 속성을 포함해야 합니다.

| 필드 | 타입 | 설명 |
|------|------|------|
| `problem_id` | UUID | 고유 식별자 |
| `title` | string | 문제 제목 (선택) |
| `content_text` | string | 문제 본문 (LaTeX 포함 텍스트) |
| `content_image_url` | string | 문제 이미지 URL (선택) |
| `answer` | string | 정답 (LaTeX 또는 텍스트) |
| `solution_steps` | JSON array | 모범 풀이 단계 |
| `subject` | enum | 과목 (대수, 기하, 미적분, 확률통계, 기타) |
| `grade` | enum | 학년 (중1~고3, 수능) |
| `difficulty` | int (1~5) | 난이도 |
| `concept_tags` | string array | 관련 개념 태그 (예: `["이차방정식", "인수분해"]`) |
| `error_type_tags` | string array | 연관 오류 유형 태그 |
| `curriculum_code` | string | 2022 개정 교육과정 코드 |
| `source` | enum | 문제 출처 (문제은행/LLM생성/교과서) |
| `created_at` | datetime | 생성 일시 |

#### 3.1.2 문제은행 규모

| 학교급 | 최소 문제 수 (초기 구축) |
|--------|------------------------|
| 중학교 (1~3학년) | 3,000문항 이상 |
| 고등학교 공통·수학I·II | 5,000문항 이상 |
| 미적분·확률통계·기하 | 3,000문항 이상 |
| **합계** | **11,000문항 이상** |

### 3.2 문제 출제 API

#### 3.2.1 개인화 문제 출제

```
GET /api/v1/problems/recommend
```

요청 파라미터:

| 파라미터 | 필수 | 설명 |
|----------|------|------|
| `user_id` | ✅ | 학습자 식별자 |
| `grade` | ✅ | 학년 |
| `subject` | ❌ | 과목 필터 |
| `difficulty_range` | ❌ | 난이도 범위 (예: `2-4`) |
| `concept_tags` | ❌ | 특정 개념 태그 필터 |
| `limit` | ❌ | 반환 문제 수 (기본 1, 최대 10) |

개인화 추천 로직 요구사항:
- 사용자의 최근 오답 개념 태그 우선 출제
- 난이도 스캐폴딩: 직전 정답 연속 횟수에 따라 난이도 자동 상향
- 동일 문제 24시간 내 재출제 금지
- 취약 개념 대비 강점 개념 출제 비율 = 7:3

#### 3.2.2 특정 조건 문제 조회

```
GET /api/v1/problems
GET /api/v1/problems/{problem_id}
```

#### 3.2.3 LLM 기반 유사 문제 동적 생성

```
POST /api/v1/problems/generate
```

요청 바디:

```json
{
  "reference_problem_id": "uuid",
  "concept_tags": ["이차방정식", "인수분해"],
  "difficulty": 3,
  "error_type": "conceptual_error",
  "grade": "고1",
  "variation_type": "number_change"  // number_change | condition_change | context_change
}
```

생성 요구사항:
- LLM(GPT-4o 또는 Claude)을 활용하여 원본 문제와 동일 개념, 다른 숫자·조건의 문제 생성
- 생성된 문제는 사람이 검토하거나 자동 검증(수식 파서로 정답 검증) 후 문제은행에 저장
- 생성 요청 후 응답 시간 **5초 이내**

### 3.3 문제 전달 방식

| 시나리오 | 방식 | 설명 |
|----------|------|------|
| 일반 문제 조회 | REST GET | 문제 목록 또는 단건 조회 |
| 실시간 세션 중 문제 출제 | WebSocket | 학습 세션 중 문제 순차 전달 |
| 오프라인 캐싱 | 앱 내 저장 | 최대 50문항 사전 다운로드 |

## 4. 펜 입력 캡처 및 전송 요구사항

### 4.1 클라이언트 캔버스 구현

#### 4.1.1 iOS (PencilKit 기반)

| 요구사항 | 상세 |
|----------|------|
| SDK | PencilKit (iOS 13+) |
| 지원 입력 | Apple Pencil 1·2세대, 손가락 터치 |
| 필압 반응 | PKStrokePoint의 `force` 값을 활용한 가변 선 굵기 |
| 팜 리젝션 | PencilKit 기본 팜 리젝션 활성화 |
| 캔버스 크기 | 문제 표시 영역 하단 전체 (최소 500pt 높이) |
| 실행취소/재실행 | 시스템 제스처 및 버튼 UI 제공 |
| 지우개 | 픽셀 지우개 및 스트로크 지우개 모두 지원 |

#### 4.1.2 Android (Ink API 기반)

| 요구사항 | 상세 |
|----------|------|
| SDK | `androidx.ink` (Jetpack Ink) 또는 Android Ink API |
| 지원 입력 | S-Pen, USI 펜, 손가락 터치 |
| 필압 반응 | `MotionEvent.AXIS_PRESSURE` 기반 가변 굵기 |
| 팜 리젝션 | `PointerType.STYLUS` 우선 처리 로직 |
| 저지연 렌더링 | 저지연 렌더링 API(`SurfaceView` 또는 `GLSurfaceView`) 활용 |

#### 4.1.3 공통 캔버스 UX 요구사항

- 펜 도구: 굵기 3단계 (얇음·보통·굵음), 색상 3종 이상 (검정·파랑·빨강)
- 지우개 도구: 스트로크 단위 지우개
- 전체 지우기 버튼
- 실행취소(Undo) / 재실행(Redo) 무제한
- 캔버스 확대/축소 (핀치 줌, 최소 1.0x, 최대 3.0x)
- 풀이 제출 버튼: 명확한 CTA 위치에 배치

### 4.2 이미지 캡처 및 전처리

#### 4.2.1 캡처 규격

| 항목 | 요구사항 |
|------|----------|
| 출력 포맷 | PNG (투명도 지원) 또는 JPEG (배경 흰색) |
| 해상도 | 최소 1080×1440 px (A4 세로 비율 기준) |
| 색상 공간 | sRGB |
| DPI | 150 DPI 이상 |
| 최대 파일 크기 | 원본 5 MB 이내, 전송 전 압축 후 2 MB 이내 |

#### 4.2.2 클라이언트 사이드 전처리

클라이언트에서 서버 전송 전 다음 처리를 수행해야 합니다.

1. **여백 자르기 (Cropping)**: 잉크 스트로크 바운딩 박스 기준 ±20px 여백 추가 후 크롭
2. **해상도 표준화**: 장기 768px (세로) 또는 1024px (가로) 기준으로 리사이징 (종횡비 유지)
3. **이미지 압축**: JPEG 품질 85% 또는 PNG 최적화
4. **Base64 인코딩**: 서버 전송 시 `data:image/png;base64,...` 형식

#### 4.2.3 스트로크 메타데이터 (선택 제공)

정밀도 향상을 위해 클라이언트는 이미지와 함께 스트로크 벡터 데이터를 선택적으로 전송할 수 있어야 합니다.

```json
{
  "strokes": [
    {
      "points": [{"x": 100, "y": 200, "pressure": 0.8, "t": 0}, ...],
      "color": "#000000",
      "width": 3.0
    }
  ],
  "canvas_width": 1080,
  "canvas_height": 1440
}
```

### 4.3 서버 전송 인터페이스

#### 4.3.1 풀이 제출 API

```
POST /api/v1/submissions
Content-Type: application/json
```

요청 바디:

```json
{
  "user_id": "string (UUID)",
  "problem_id": "string (UUID)",
  "session_id": "string (UUID)",
  "image_base64": "string",
  "image_mime_type": "image/png | image/jpeg",
  "stroke_data": { ... },       // 선택
  "client_timestamp": "ISO8601",
  "device_info": {
    "platform": "ios | android",
    "os_version": "string",
    "input_type": "stylus | touch"
  }
}
```

응답:

```json
{
  "submission_id": "string (UUID)",
  "status": "received | processing | completed | failed",
  "estimated_processing_time_ms": 3000
}
```

#### 4.3.2 처리 결과 조회 (폴링)

```
GET /api/v1/submissions/{submission_id}/status
```

#### 4.3.3 실시간 상태 수신 (WebSocket)

```
WS /ws/v1/submissions/{submission_id}
```

서버는 처리 단계별로 다음 메시지를 푸시합니다.

| 단계 | 메시지 타입 |
|------|------------|
| 이미지 수신 완료 | `{ "stage": "received" }` |
| 전처리 완료 | `{ "stage": "preprocessed" }` |
| HTR 인식 완료 | `{ "stage": "htr_done", "latex": "..." }` |
| 분석 완료 | `{ "stage": "analysis_done", "result_id": "..." }` |

## 5. AI 인식 엔진 연동 요구사항

### 5.1 서버 사이드 이미지 전처리

서버에서 HTR 엔진에 전달하기 전 다음 전처리를 수행합니다.

| 단계 | 처리 내용 | 기술 옵션 |
|------|-----------|-----------|
| 1. 이진화 | 배경/잉크 분리, Otsu's Threshold | OpenCV |
| 2. 노이즈 제거 | 가우시안 블러, 모폴로지 연산 | OpenCV |
| 3. 기울기 보정 | 필기 라인 기울기 검출 후 회전 | OpenCV, pytesseract |
| 4. 행 분리 | 풀이 행 단위 분리 (Horizontal Projection Profile) | 커스텀 구현 |
| 5. 수식/텍스트 영역 분류 | 수식 영역과 자연어 설명 영역 분리 | YOLO 또는 규칙 기반 |

### 5.2 HTR 엔진 요구사항

#### 5.2.1 수식 인식 (Math HTR)

| 요구사항 | 상세 |
|----------|------|
| 출력 포맷 | LaTeX (예: `\frac{x^2 + 1}{2x - 3}`) |
| 지원 기호 범위 | 사칙연산, 분수, 루트, 거듭제곱, 극한, 미적분, 행렬, 집합, 확률 기호 |
| 인식 정확도 | 95% 이상 (Character Error Rate 기준, 표준 필기) |
| 응답 시간 | 단일 이미지 기준 **2초 이내** |
| 구현 옵션 | MyScript Math SDK, MathPix API, Pix2Tex (오픈소스), 자체 파인튜닝 모델 |

제안사는 선택한 HTR 엔진에 대해 다음을 명시해야 합니다.
- 라이선스 비용 및 API 호출 단가
- 지원 수학 기호 범위 (LaTeX 표준 기준)
- 오프라인(온디바이스) 처리 가능 여부

#### 5.2.2 텍스트 인식 (일반 HTR)

한글·영문 손글씨 텍스트 인식(풀이 설명 부분):

| 요구사항 | 상세 |
|----------|------|
| 지원 언어 | 한국어, 영어 |
| 구현 옵션 | Google ML Kit (온디바이스), Apple Vision Framework (iOS), Naver Clova OCR |
| 인식 정확도 | 90% 이상 (한국어 손글씨 기준) |

### 5.3 LaTeX 후처리 및 구조화

HTR 엔진의 원시 LaTeX 출력에 대해 다음 후처리를 수행합니다.

1. **LaTeX 파싱**: 수식 구조를 Abstract Syntax Tree(AST)로 변환
2. **단계 분리**: 등호(`=`), 줄바꿈 패턴 기반 풀이 단계 분리
3. **수식 검증**: SymPy 또는 Mathematica를 통해 수식 문법 유효성 확인
4. **정답 비교**: 제출 답안 LaTeX와 문제 정답 LaTeX의 수학적 동치 여부 판별

```python
# 수식 동치 판별 예시 (SymPy 활용)
from sympy.parsing.latex import parse_latex
from sympy import simplify

student_expr = parse_latex(student_latex)
correct_expr = parse_latex(correct_latex)
is_correct = simplify(student_expr - correct_expr) == 0
```

### 5.4 멀티모달 LLM 연동

HTR 결과(LaTeX 텍스트) 및 원본 이미지를 함께 LLM에 전달하여 풀이를 해석합니다.

```
POST /api/v1/analyze
Content-Type: application/json

{
  "submission_id": "uuid",
  "problem_id": "uuid",
  "htr_result": {
    "latex": "...",
    "steps": ["step1_latex", "step2_latex", ...],
    "confidence": 0.97
  },
  "image_base64": "...",    // 멀티모달 LLM에 원본 이미지 직접 전달
  "user_id": "uuid"
}
```

LLM 프롬프트 구성 요구사항:
- 문제 원문 포함
- 학습자 학년 및 이전 오류 유형 히스토리 포함 (시스템 프롬프트)
- 인식된 LaTeX 풀이 단계 포함
- 원본 손글씨 이미지 동시 제공 (이미지 인식 불확실 구간 보완)
